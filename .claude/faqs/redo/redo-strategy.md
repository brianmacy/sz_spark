# Redo processing strategy

## What is redo and does it run automatically?

When an operation changes an entity's resolution state, Senzing enqueues **redo records** for affected
neighbors. Redo is **never processed automatically** — you call `getRedoRecord()` to dequeue one and
`processRedoRecord(redo, SZ_WITH_INFO)` to process it. Skipping redo leaves entity resolution
incomplete.

Two facts shape the Spark design:

- **`getRedoRecord() == null` is NOT "done."** The queue refills continuously — ongoing loads enqueue
  redo, and processing redo generates more redo. The canonical Senzing worker, on an empty queue,
  **pauses (~30s) and resumes**. So redo is a **recurring/scheduled** activity, not a one-shot drain.
- **Processing should be parallelized**, like the other operations. The documented patterns
  (`RedoContinuousViaFutures`, the redo-worker-pool) use a **single dequeuer** feeding **many parallel
  processors** — never concurrent multi-consumer `getRedoRecord()`.
- Never use `countRedoRecords()` as a loop condition (full table scan; monitoring only).

## The chosen approach (primary): a parallel, scheduled RedoJob with getRedoRecord() as the source

Model redo as **its own Spark job, just like AddUpdate/Delete/Search**, but with `getRedoRecord()` as
the **source** instead of input files:

1. **Scheduled / recurring** (Databricks Job / Airflow / cron) — because the queue keeps refilling, each
   run drains what is currently pending; the schedule is what makes it continuous. (Equivalently, a
   long-lived job that pauses and resumes on empty.)
2. **Single dequeuer builds the DataFrame:** a driver-side loop calls `getRedoRecord()` to pull a batch
   of redo-record JSON strings into a DataFrame (one consumer of the queue — matches the canonical
   single-`getRedoRecord()`-thread pattern and sidesteps the undemonstrated concurrent-dequeue
   question).
3. **Process in parallel:** `repartition` the DataFrame randomly, then `mapPartitions` → each task,
   single-threaded against the **shared per-executor engine**, calls
   `processRedoRecord(redoString, SZ_WITH_INFO)` and emits `AFFECTED_ENTITIES` (`op=REDO`) into the
   **same** affected-entities sink the loaders use. Same engine lifecycle, error taxonomy, LONG_RECORD,
   and progress logging as the other jobs.
4. **Monitoring:** `countRedoRecords()` only as a coarse health gauge, never in a loop.

**Durability note:** `getRedoRecord()` *removes* the record from the queue, so a dequeued record now
lives only in the DataFrame. Spark **task** retries are safe (the materialized source DataFrame is
replayed), but a **whole-job** crash after dequeue can lose the in-flight batch. Keep per-run batch
sizes modest; redo is convergent (a lost redo is usually regenerated by later activity), and the
recurring schedule re-drains. For strict durability, persist the pulled batch before processing.

## Rejected alternatives (do not re-propose)

- **Per-task end-drain** — draining redo at the end of every add/update/delete subpartition. Makes
  loader task timing unpredictable and relies on concurrent multi-consumer `getRedoRecord()` across
  executors, which the Senzing docs do not demonstrate.
- **Single-dequeuer one-shot drain** — one consumer draining to empty once. Does not parallelize
  processing, and a one-shot drain ignores that the queue refills.
